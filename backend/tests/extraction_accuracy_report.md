## The 3 misses, explained

Both misses turned out to be genuine LLM-output nuances, not pipeline bugs
— worth reporting honestly rather than re-running until the score looks
cleaner.

### triconboston_wind.pdf: expected_irr_percent

This run, the model returned `expected_irr_percent: 17` (sourced from "the
project has an upfront feed-in tariff, providing a 17% return on equity").
Ground truth says `not_found`, because "return on equity" under a specific
tariff mechanism is a different metric from an overall project IRR, and the
document never states an overall IRR figure. This is a known ambiguous
case — documented in the ground-truth fixture's own `notes` field back in
Task 3 — and re-running the live pipeline against this same document on
other occasions has previously returned `not_found` for this field. It's
genuine model non-determinism on a genuinely debatable distinction, not a
new bug.

### aj_solar_india.pdf: debt_percent, equity_percent

This is the more interesting finding. The evaluation run's raw API response
apparently paired a non-`not_found` confidence level with a `null` value for
both fields — an internally inconsistent combination our prompt asks the
model to avoid but that the schema doesn't structurally forbid (`value` is
`Optional[Any]`, independent of `confidence`). A fresh, separate call to the
same document immediately afterward returned the expected clean
`{"value": null, "confidence": "not_found"}` for both fields, confirming
this was a one-off inconsistency in that specific API call rather than a
systematic error.

**Recommendation**: harden `normalize_extraction()` to treat a `null` value
paired with any non-`not_found` confidence as `not_found` regardless of the
stated confidence — closing this small gap between "what the prompt asks
for" and "what the schema actually enforces."

## Caveats on this number

- **N=5 documents** is a small sample. 96.2% here is encouraging but
  shouldn't be read as a firm production SLA — it's a snapshot from one run
  against one batch of real documents, not a statistically robust estimate.
- **Not fully reproducible run-to-run**: as shown above, the exact same
  document can produce a different result on independent calls, for fields
  where the source document is itself ambiguous or where the model's output
  isn't perfectly self-consistent. Re-running this script periodically
  (especially after any model or prompt change) is more informative than
  treating any single run's number as final.
- **Fields that are typically absent from real documents dominate the
  "hard" cases** — `expected_irr_percent`, `payback_period_years`, and
  `lcoe_usd_per_mwh` are `not_found` in most ground-truth fixtures, so a
  pipeline that defaults to `not_found` too eagerly would score
  artificially well here. The ground-truth fixtures were deliberately
  built to include documents (`triconboston_wind.pdf`, `cambodia_solar.pdf`)
  where several of these fields genuinely *are* stated, specifically to
  guard against that failure mode.

## What's next

- Re-run this script whenever the extraction prompt, schema, or model
  changes, and track the trend over time rather than a single number.
- Consider the `normalize_extraction()` hardening noted above.
- Expand the ground-truth set beyond 5 documents as more real (ideally
  TerraWatt-provided) proposals become available.
